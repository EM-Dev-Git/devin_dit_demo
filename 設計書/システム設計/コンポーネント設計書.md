# コンポーネント設計書

## 1. 概要

### 目的
トランスクリプトから議事録作成APIシステムの各コンポーネントの詳細設計を定義し、実装時の具体的な仕様を明確化する

### 対象範囲
- Routers Layer コンポーネント
- Modules Layer コンポーネント
- Schemas Layer コンポーネント
- コンポーネント間インターフェース

### 前提条件
- FastAPI フレームワークの使用
- 3層アーキテクチャの採用
- 依存性注入パターンの実装

## 2. 設計方針

### 基本方針
- **単一責任原則**: 各コンポーネントは明確な責務を持つ
- **疎結合**: コンポーネント間の依存関係を最小化
- **高凝集**: 関連する機能を適切にグループ化
- **再利用性**: 共通機能の再利用を促進

### 制約事項
- FastAPI の機能制限
- Python の言語制約
- SQLite データベースの制限

### 品質要件
- **保守性**: 理解しやすく変更しやすい設計
- **テスタビリティ**: 単体テストが容易な設計
- **拡張性**: 新機能追加に対応可能

## 3. Routers Layer コンポーネント

### 3.1 Auth Router (auth.py)

#### 責務
- 認証関連のHTTPエンドポイント提供
- リクエスト/レスポンスの処理
- 認証ビジネスロジックの呼び出し

#### エンドポイント定義
```python
@router.post("/register", response_model=UserResponse)
async def register_user(user: UserCreate, db: Session = Depends(get_db)):
    """ユーザー登録エンドポイント"""
    pass

@router.post("/login", response_model=Token)
async def login_user(user_credentials: UserLogin, db: Session = Depends(get_db)):
    """ユーザーログインエンドポイント"""
    pass

@router.post("/refresh", response_model=Token)
async def refresh_token(current_user: User = Depends(get_current_user)):
    """トークンリフレッシュエンドポイント"""
    pass
```

#### 依存関係
- **入力**: Pydantic スキーマ (UserCreate, UserLogin)
- **出力**: Pydantic スキーマ (UserResponse, Token)
- **依存**: Auth Handler, Database Session

#### エラーハンドリング
- バリデーションエラー (422)
- 認証エラー (401)
- 重複エラー (409)
- サーバーエラー (500)

## 7. 実装考慮事項

### 開発時の注意点
- **依存性注入**: FastAPI の Depends を適切に使用
- **型ヒント**: すべての関数に適切な型ヒントを付与
- **エラーハンドリング**: 適切な HTTP ステータスコードの返却
- **ログ出力**: 個人情報を含まないログ設計

### 既知の課題
- OpenAI API の利用制限による処理時間の変動
- SQLite の同時書き込み制限
- JWT トークンの無効化機能未実装

### 代替案
- **データベース**: PostgreSQL への移行検討
- **認証**: OAuth 2.0 対応
- **AI連携**: 複数AIモデルの選択機能
