# 総合テスト設計書 - transcript_minutes_api2

**対象システム**: transcript_minutes_api2  
**作成日**: 2025年6月24日

## テスト環境

### システム構成
```yaml
環境: 本番相当環境
データベース: SQLite (実ファイル)
外部API: 
  - OpenAI API (実接続・レート制限考慮)
  - Microsoft Graph API (実接続・認証設定)
認証: JWT (実トークン生成・検証)
ログ: 実ファイル出力 (app.log)
```

### 前提条件
- OpenAI APIキーが有効であること
- Microsoft Graph API認証情報が設定済みであること
- インターネット接続が安定していること
- テスト用Microsoft 365アカウントが利用可能であること

## 総合テストシナリオ

### シナリオ1: 新規ユーザーの完全ワークフロー

#### 1.1 ユーザー登録から議事録生成まで
**目的**: 新規ユーザーが初回利用時に期待する一連の操作が正常に完了すること

**テストステップ**:
1. **ユーザー登録**
   - POST /auth/register で新規ユーザー作成
   - 有効なメールアドレス・パスワードで登録
   - 登録完了レスポンスの確認

2. **ログイン**
   - POST /auth/login でJWTトークン取得
   - トークンの有効性確認

3. **プロフィール確認**
   - GET /users/profile で登録情報確認
   - 表示内容の正確性検証

4. **議事録生成（手動入力）**
   - POST /minutes/generate で議事録生成
   - 日本語トランスクリプトでの生成テスト
   - 生成内容の品質確認

5. **履歴確認**
   - GET /minutes/history で生成履歴確認
   - GET /minutes/{id} で詳細確認

**期待結果**:
- 全ステップが5分以内に完了
- 生成された議事録が読みやすく構造化されている
- エラーが発生しない

#### 1.2 Microsoft Graph連携ワークフロー
**目的**: Teams会議のトランスクリプトから議事録生成までの完全フロー

**テストステップ**:
1. **会議一覧取得**
   - GET /graph/meetings/{user_id} でTeams会議一覧取得
   - 最近の会議が正しく表示されること

2. **トランスクリプト一覧取得**
   - GET /graph/meetings/{user_id}/{meeting_id}/transcripts
   - 利用可能なトランスクリプトが表示されること

3. **トランスクリプト内容取得**
   - GET /graph/meetings/{user_id}/{meeting_id}/transcripts/{transcript_id}/content
   - 実際の会議内容が取得できること

4. **Graph経由議事録生成**
   - POST /graph/minutes/generate
   - Teams会議から高品質な議事録が生成されること

**期待結果**:
- Graph API連携が正常に動作
- 実際の会議内容から意味のある議事録が生成される
- 生成時間が10分以内

### シナリオ2: 継続利用ユーザーの日常業務フロー

#### 2.1 複数議事録管理
**目的**: 日常的に複数の会議議事録を管理するユーザーの使用パターン

**テストステップ**:
1. **複数議事録の連続生成**
   - 異なる会議内容で5件の議事録を連続生成
   - 各議事録の品質確認

2. **履歴管理**
   - GET /minutes/history でページネーション動作確認
   - 日付順ソートの確認

3. **プロフィール更新**
   - PUT /users/profile でユーザー情報更新
   - 更新後の整合性確認

4. **トークンリフレッシュ**
   - POST /auth/refresh で長時間利用シミュレーション

**期待結果**:
- 複数議事録の管理が効率的に行える
- データの整合性が保たれる
- セッション管理が適切に動作

### シナリオ3: 高負荷・ストレステスト

#### 3.1 同時アクセステスト
**目的**: 複数ユーザーの同時利用時の安定性確認

**テストステップ**:
1. **同時ユーザー登録**
   - 10ユーザーの同時登録処理
   - データベース競合状態の確認

2. **同時議事録生成**
   - 5ユーザーによる同時議事録生成
   - OpenAI APIレート制限への対応確認

3. **データベース負荷テスト**
   - 大量データ投入後の検索性能確認
   - 100件以上の議事録履歴での動作確認

**期待結果**:
- 同時アクセス時もエラーが発生しない
- レスポンス時間が許容範囲内（5秒以内）
- データの整合性が保たれる

#### 3.2 大容量データテスト
**目的**: 大きなトランスクリプトデータでの動作確認

**テストステップ**:
1. **大容量トランスクリプト処理**
   - 10,000文字以上のトランスクリプトで議事録生成
   - メモリ使用量の監視

2. **長時間会議データ**
   - 2時間以上の会議トランスクリプトでの処理
   - タイムアウト処理の確認

**期待結果**:
- 大容量データでも正常に処理される
- メモリリークが発生しない
- 適切なエラーハンドリングが動作

### シナリオ4: セキュリティ・認証テスト

#### 4.1 認証セキュリティ
**目的**: 認証機能のセキュリティ強度確認

**テストステップ**:
1. **不正アクセス試行**
   - 無効なJWTトークンでのアクセス試行
   - 期限切れトークンでのアクセス試行
   - 他ユーザーのデータへの不正アクセス試行

2. **パスワードセキュリティ**
   - 弱いパスワードでの登録試行
   - パスワードハッシュ化の確認

3. **セッション管理**
   - 複数デバイスからの同時ログイン
   - ログアウト後のトークン無効化確認

**期待結果**:
- 不正アクセスが適切にブロックされる
- セキュリティエラーが適切に返される
- ユーザーデータが保護される

#### 4.2 データプライバシー
**目的**: ユーザーデータの適切な保護確認

**テストステップ**:
1. **データ分離**
   - ユーザーAがユーザーBの議事録にアクセスできないこと
   - 管理者権限なしでの他ユーザーデータアクセス制限

2. **ログ情報**
   - 機密情報がログに出力されていないこと
   - パスワードやトークンの平文記録がないこと

**期待結果**:
- ユーザーデータが適切に分離されている
- 機密情報の漏洩がない

- ユーザーに分かりやすいエラー情報が提供される

#### 5.2 データベース障害対応
**目的**: データベース関連の障害時の動作確認

**テストステップ**:
1. **データベース接続障害**
   - DB接続不可時の動作確認
   - 接続復旧後の自動回復確認

2. **データ整合性エラー**
   - 不正なデータ投入時の動作確認
   - トランザクション失敗時のロールバック確認

**期待結果**:
- データベース障害が適切にハンドリングされる
- データの整合性が保たれる

### シナリオ6: ユーザビリティ・品質テスト

#### 6.1 議事録品質評価
**目的**: 生成される議事録の品質が実用レベルであることの確認
